# Technical Review: Guild

## Date: 2026-04-03

## Project Overview

Guild is a browser-based medieval fantasy city management simulation built with Vue 3, TypeScript 5.8, Vite, and Tailwind CSS v4. The project is at prototype stage: a single-screen city dashboard with tick-based simulation, three production buildings (Lumber Mill, Iron Mine, BlackSmith), and a skeleton adventurer/equipment domain that isn't yet player-facing.

The codebase targets a layered architecture: a pure-TypeScript game domain, a Vue presentation layer, and an inventory accounting module. Comprehensive specs exist in `.specs/` documenting both current state and intended evolution across five roadmap phases.

## Executive Summary

The project has a solid vision and well-documented specs, but the implementation has several critical logic bugs that prevent the economic simulation from functioning correctly. The build fails due to 8 TypeScript errors, the Lumber Mill production chain is broken at step one, the Iron Mine never deposits ore into inventory, the BlackSmith bypasses the transaction system entirely, and the `validateLedger` function always returns `true` regardless of input. These issues must be resolved before any feature expansion can proceed safely.

## Critical Issues

### 1. Build Fails: 8 TypeScript Errors

The project does not compile. The root causes are:

**`Weapon` and `Armor` use `implements Partial<IWeapon>` / `implements Partial<IArmor>`** — This is the core type design flaw. By marking the interface as `Partial`, TypeScript no longer requires the abstract base classes to declare `type`, `dual_handed`, `can_dual_wield` (Weapon) or `type` (Armor). The concrete subclasses define these properties, but `Adventurer.ts` references them through the abstract base type, causing property-not-found errors.

**Fix:** Remove `Partial` from the implements clauses. Declare the missing properties as `abstract` on the base classes:

```typescript
// Weapon.ts
export abstract class Weapon extends EquippableItem implements IWeapon {
    public abstract readonly type: WeaponType;
    public abstract readonly range: number;
    public abstract readonly dual_handed: boolean;
    public abstract readonly can_dual_wield: boolean;
    // ...
}
```

**`EquippableItem.getGood()` return type mismatch** — The static method casts `this` (the class constructor) to `EquippableItem` (an instance type) via `unknown`. The `base_class` field in `IGood` expects `EquippableItem | undefined` but gets a constructor function. This needs a cleaner factory pattern or the `IGood.base_class` type needs to accept constructor references.

**`Layout.vue` incorrect import** — Uses `import City from ...` (default import) but `City` is a named export. Change to `import { City } from ...`.

**`BuildingsList.vue` prop typing** — The `activeBuildingId` prop uses `Number` constructor as its type, but should be typed as `BuildingID | null` via a proper PropType cast.

### 2. Lumber Mill Production Chain Is Broken

`TakeDownTreeAction` (the first step in the lumber cycle) is configured with:

```typescript
input = new Map([[GoodID.WoodPlank, 1]]);  // requires 1 WoodPlank to chop a tree
```

This means the Lumber Mill needs WoodPlanks to produce Lumber, but WoodPlanks are made FROM Lumber. The chain can never bootstrap. Additionally, `TakeDownTreeAction.finished()` is empty — it never produces any Lumber output. The spec says this action should produce 1 Lumber with no input requirements.

**Fix:** Remove the input requirement and add output production:

```typescript
class TakeDownTreeAction extends Action {
    total_ticks = 6;
    building_id = BuildingID.LumberMill;
    input = null;  // no input needed to chop a tree

    protected finished() {
        super.finished();
        inventoryRepository.putGood(BuildingID.LumberMill, GoodID.Lumber, 1);
    }
}
```

### 3. Iron Mine Never Deposits Ore

`MineOres` defines `output` and `output_destination` but no `input`. Since the `Action.start()` method only creates a transaction when `this.input` is set, no transaction is ever created. And since `Action.finished()` only commits a transaction when `this.transaction_id` exists, the output goods are never deposited. The commented-out line `// InventoryService.putGood(...)` in `MineOres.finished()` confirms this was noticed but not resolved.

**Fix:** Either implement a production-only transaction flow (no input, just output), or directly put goods in `finished()` like the BlackSmith does.

### 4. `validateLedger` Always Returns True

In `inventory.repository.ts`, the validation logic uses `forEach` with early `return false` statements:

```typescript
public validateLedger(id: InventoryID, ledger: GoodLedger): boolean {
    const account_ledger = this.getCountByGoodId(id);
    ledger.forEach((required_amount, good_id) => {
        // ...
        if (required_amount > account_amount) {
            return false;  // returns from the forEach callback, NOT from validateLedger
        }
    });
    return true;  // always reached
}
```

The `return false` inside `forEach` only exits the callback, not the outer function. This means validation always passes, and `takeGoods` will happily subtract goods into negative counts. Every transaction and inventory check in the game is silently broken.

**Fix:** Use a `for...of` loop or convert to `Array.from(ledger.entries()).every(...)`.

### 5. BlackSmith Bypasses the Transaction System

`MakeIngotAction` and all `BaseMakeAction` subclasses call `inventoryRepository.putGood()` directly in their `finished()` methods, but they never consume their input ingredients. The actions define `input` maps (e.g., `[GoodID.IronIngot, 2]` for a sword), and `chooseNextAction` calls `validateInput()` and `validateLedger()`, but:

- `MakeIngotAction.finished()` produces 1 ingot but never removes the 2 IronOre
- `BaseMakeAction.finished()` produces equipment but never removes the IronIngot inputs
- Neither calls `super.finished()` to commit any transaction

The result: the BlackSmith has infinite resources. Its 400 starting ore is never consumed.

**Fix:** Either configure `input_origin`/`output_destination` properly and call `super.finished()`, or explicitly call `inventoryRepository.takeGoods()` before producing output.

## Architecture Recommendations

### Decouple Actions from Singletons

Actions currently import and call `gameController` and `inventoryRepository` singletons directly. This creates hidden dependencies and makes individual actions impossible to test in isolation. Consider passing the inventory repository (or a scoped service) into actions via the building or a context object:

```typescript
protected chooseNextAction(ctx: BuildingContext): Action { ... }
```

### Unify the Two Inventory Systems

The codebase has two separate inventory implementations:

1. `src/modules/inventory/` — the repository/service/transaction system used by buildings
2. `src/game/common/Inventory.ts` — a standalone `Inventory` class used by `City` and `Adventurer`

The specs note this split. The repository system is the more capable design (supports transactions, equipment, accounts). The legacy `Inventory` class should be phased out and adventurers migrated to use account-based inventory, or at minimum the two should share a common interface.

### Establish a Transaction Contract

The current transaction system has three different usage patterns:
- LumberMill's `MakeWoodAction`: properly configures input/output and relies on `Action.start()` + `Action.finished()`
- BlackSmith: bypasses transactions entirely, directly mutating the repository
- IronMine: configures output but never materializes it

Pick one pattern and standardize. The `Action` base class should enforce or clearly document which lifecycle hooks are responsible for inventory mutations.

### Remove Debug Logging

There are `console.log('[Module] Loaded')` and `console.log('[Module] OK')` calls throughout — in `City.ts`, `Building.ts`, `Armor.ts`, `Weapon.ts`, `EquippableItem.ts`, `InventoryRepository.ts`, etc. These are initialization-time debug traces. Consider removing them or gating behind a debug flag before the codebase grows.

## Data Structure & Performance

### Equipment Placeholder Values

Every armor piece has identical stats (`value: 10`, `base_armor_value: 10`, `weight: 5`) and every weapon shares the same base damage (`base_damage_value: 10`, `weight: 5`). The specs note these are placeholders, but this means the durability degradation system (which is actually well-implemented with the linear-to-exponential curve) has no meaningful differentiation to work with. Before adding adventurer gameplay, these values need to be differentiated or the equipment tiers won't matter.

### `damage_value` Computed Once in Constructor

In `Weapon.ts`, `damage_value` is set in the constructor:

```typescript
constructor(wear: number = 0) {
    super(wear);
    this.damage_value = this.damageValue;  // computed once, never updates
}
```

If wear increases over time (as the durability system implies), `damage_value` will be stale. It should be a getter, not a stored property. Same issue exists in `Armor` with `armor_value`.

### Production Priority Object Uses String Keys

In `BlackSmith.chooseNextAction()`, the production priority list is a plain object with `GoodID` enum values as keys:

```typescript
const production_list = {
    [GoodID.IronSword]: { desired_amount: 12, action: MakeIronSwordAction },
    // ...
};
```

Then iterated with `Object.entries()`, which returns string keys. The comparison `list.get(good_id as GoodID)` works because `GoodID` values are strings, but this is fragile — if `GoodID` were ever changed to numeric enums, this would silently break. Consider using a `Map<GoodID, ...>` or an array of tuples to preserve type safety.

## TypeScript Improvements

### `implements Partial<T>` Is an Antipattern

As discussed in Critical Issue #1, using `Partial` on an implements clause defeats the purpose of the interface contract. The abstract classes should declare all interface properties as abstract, and concrete classes implement them.

### Missing Type Safety on Action Inputs

Action `input` and `output` are typed as `null | GoodLedger` with no compile-time guarantee that a production action has its required fields set. Consider using discriminated union types or builder patterns:

```typescript
type ProductionConfig =
    | { type: 'gather'; output: GoodLedger }
    | { type: 'transform'; input: GoodLedger; output: GoodLedger }
    | { type: 'transport'; input: GoodLedger };
```

### Stale `IronSword` in Goods Map

In `AvailableGoods.ts`, `GoodID.IronSword` is mapped as a stackable good entry (value 1, weight 1) alongside the equipment-class definition. This dual representation (both a simple good and an equippable item) can cause confusion in the inventory system — `getCount` behaves differently depending on whether `base_class` is set.

### `building_id` on Actions Is Error-Prone

Every action subclass must manually set `building_id = BuildingID.X`. If forgotten, `getBuilding()` will crash at runtime. Consider passing the building reference (or ID) into the action constructor from the building that creates it.

## Vue.js Patterns

### Game State Wrapped in `reactive()` at App Level

`App.vue` wraps the `GameController` singleton in `reactive()`. This works for the prototype but will cause increasing overhead as the game state grows — every tick will trigger Vue's proxy traps on every property access. For now this is fine, but as entity counts grow, consider using `shallowReactive` or `markRaw` for high-frequency game data and only making UI-facing derived state reactive.

### Building Detail Is a Raw `<pre>` Dump

The spec notes this: the selected building view is currently `<pre>{{ activeBuilding }}</pre>`. This is appropriate for prototyping but should be replaced with a dedicated component before the UI milestone.

### Component Structure Is Minimal

Only two custom components exist (`Button.vue`, `BuildingsList.vue`). The architecture spec correctly identifies that this will need to expand. As building detail panels, adventurer rosters, and inventory views are added, extracting composables (`useGameLoop`, `useBuilding`, `useInventory`) will prevent `App.vue` from becoming monolithic.

## What's Working Well

**Spec documentation is excellent.** The `.specs/` directory is thorough, honest about gaps, and clearly distinguishes between implemented behavior and intended behavior. This is rare for a prototype and will pay dividends as the project grows.

**The domain model design is sound.** The conceptual layering — city as aggregate root, buildings as production subsystems, workers executing time-based actions, adventurers as equipment consumers — is well thought out. The class hierarchy for equipment (abstract `EquippableItem` → `Weapon`/`Armor` → concrete items) is a reasonable foundation.

**The durability degradation formula is clever.** The linear-to-exponential degradation curve in `EquippableItem.degradationMultiplier` is a nice game design touch that will make equipment repair decisions interesting.

**The BlackSmith's priority production system is well-designed.** The ordered production list with minimum stock targets and fallback behavior is a clean way to model automated crafting priorities.

**The transaction system concept is right.** The idea of reserving goods on action start and committing on finish is the correct pattern for a tick-based economy — it prevents double-spending across concurrent actions. The implementation just needs to be consistently applied.

## Suggested Next Steps

### Priority 1: Fix the Build (Effort: 1-2 hours)

1. Remove `Partial` from `Weapon implements` and `Armor implements`, add abstract property declarations
2. Fix `EquippableItem.getGood()` typing
3. Fix `Layout.vue` import and `BuildingsList.vue` prop types
4. Verify `npm run build` passes clean

### Priority 2: Fix the Economy Logic (Effort: 2-4 hours)

1. Fix `validateLedger` to actually validate (use `for...of` instead of `forEach`)
2. Fix `TakeDownTreeAction` — remove WoodPlank input, add Lumber output
3. Fix `MineOres` — ensure ore is deposited into inventory on finish
4. Fix BlackSmith actions — ensure inputs are consumed when outputs are produced
5. Verify a full production cycle: tree → lumber → planks → sell, ore → sell, ore → ingot → equipment

### Priority 3: Connect Building Money to City (Effort: 30 min)

Sales revenue currently accumulates in `building.money` but never flows to `city.money`, which is what the UI header displays. Add a mechanism (city tax, direct transfer, or aggregation) so the player can see economic progress.

### Priority 4: Standardize the Action/Transaction Pattern (Effort: 2-3 hours)

Pick one consistent approach for how actions interact with inventory. Document it. Refactor all three buildings to follow the same pattern. This is essential foundation work before adding more buildings.

### Priority 5: Replace Placeholder Equipment Stats (Effort: 1 hour)

Differentiate weapon damage, armor values, weights, and costs across material tiers (wood < leather < iron) and slot types. The durability system is ready but needs real numbers to be meaningful.
